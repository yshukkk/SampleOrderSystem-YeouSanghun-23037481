# PRD — SampleOrderSystem (반도체 시료 생산주문관리 시스템)

Source: `Project.pdf` (Chapter 01 배경/시스템 개요, Chapter 02 기능 명세). Read the PDF directly if any rule below seems ambiguous — this document distills it but the PDF is the source of truth.

## 배경

가상의 반도체 회사 "S-Semi"는 다양한 반도체 시료(Sample)를 생산해 연구소·팹리스 업체·대학 연구실 등에 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다. 기존에는 엑셀/메모장으로 주문을 관리해 실수가 잦고 재고·공정 현황 파악이 어려웠다. 이를 대체하는 콘솔 기반 관리 시스템을 만든다.

## 역할

- **고객**: 필요한 시료를 이메일 등으로 요청(시스템 밖 프로세스).
- **주문 담당자**: 고객 요청에 맞게 주문서를 작성(=시스템에 주문 등록). 이 시스템의 1차 사용자.
- **생산 담당자**: 개발 시료를 시스템에 등록하고, 주문 수신 후 승인/거절 처리. 승인/거절 및 생산·출고 관리도 담당.

시스템은 이 두 담당자(주문/생산)가 콘솔로 직접 명령을 입력해 조작하는 단일 콘솔 애플리케이션이다(역할별 로그인 분리는 명세에 없음 — 메뉴로 기능이 구분될 뿐).

## 도메인 모델

### 시료 (Sample)
| 속성 | 설명 |
|---|---|
| 시료 ID | 고유 식별자 |
| 이름 | 시료명 |
| 평균 생산시간 | 1개 생산에 걸리는 평균 시간 |
| 수율 (yield) | 정상적인 시료 수 / 총 생산 시료 수 (예: 100개 중 90개 정상 → 0.9) |
| 현재 재고 | 즉시 출고 가능한 수량 |

시스템에 등록된 시료만 주문 가능.

### 주문 (Order)
| 속성 | 설명 |
|---|---|
| 주문번호 | 고유 식별자 |
| 시료 ID | 주문 대상 시료 |
| 고객명 | 주문자 |
| 주문 수량 | 요청 수량 |
| 상태 | 아래 상태 표 참조 |

### 주문 상태 (State Machine)

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수(예약) 직후 |
| REJECTED | 주문 거절. **정상 흐름 밖의 상태이며 모니터링에서 제외**한다 |
| PRODUCING | 승인 완료 + 재고 부족 → 생산 중 |
| CONFIRMED | 승인 완료 + 재고 충분 → 출고 대기 중 |
| RELEASED | 출고 완료 |

전이:
```
RESERVED --거절--> REJECTED
RESERVED --승인, 재고충분--> CONFIRMED --출고 처리--> RELEASED
RESERVED --승인, 재고부족--> PRODUCING --생산 완료(자동)--> CONFIRMED --출고 처리--> RELEASED
```

### 생산 라인 (Production Line)

- 공장에서 시료 하나를 생산하는 설비 흐름. 하나의 생산 라인은 시료를 **하나씩** 생산(동시에 여러 개 생산 불가 — 단일 라인 가정).
- 주문이 들어온(승인되어 재고 부족한) 시료에 대해서만 생산.
- 생산 대기열은 **생산 큐**이며 **FIFO**로 스케줄링된다(먼저 승인되어 큐에 들어온 주문부터 처리).
- 계산식:
  - 부족분 = 주문 수량 − 현재 재고
  - 실 생산량 = `ceil(부족분 / 수율)` (수율에 의한 불량을 보정해 여유 있게 생산)
  - 총 생산 시간 = 평균 생산시간 × 실 생산량
- 생산이 완료되면 해당 주문 상태가 `PRODUCING` → `CONFIRMED`로 전환되고, 생산된 수량이 재고에 반영된다.

## 기능 명세

### 메인 메뉴

기능별 선택 화면을 표시하고, 전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 등)를 함께 보여준다.

| 메뉴 | 의미 |
|---|---|
| 시료 관리 | 새로운 시료 등록, 목록 조회, 이름 검색 |
| 주문 (접수/승인/거절) | 고객 주문 접수 및 생산 라인 담당자의 승인·거절 처리 |
| 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 출고 처리 | CONFIRMED 상태 주문에 대해 출고 실행 |
| 생산 라인 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |

화면 레이아웃/서식은 PDF의 예시 UI를 참고만 하고 자유롭게 구성 가능(명세는 필수, 화면 디자인은 자율).

### 1. 시료 관리

- **시료 등록**: 시료 ID, 이름, 평균 생산시간, 수율을 이 순서(ID → 이름 → 평균 생산시간 → 수율)로 입력받아 신규 시료 추가. 시료 ID는 자동 부여가 아니라 사용자가 직접 지정하는 값이며, 이미 존재하는 ID를 다시 등록하려 하면 기존 레코드를 덮어쓰지 않고 오류로 거부한다. 재고 초기값은 0으로 시작(별도 초기 재고 입력은 명세에 없음).
- **시료 조회**: 등록된 모든 시료 목록 + 현재 재고 수량을 함께 표시.
- **시료 검색**: `ID: <숫자>`(정확한 ID 조회) 또는 `이름: <검색어>`(이름 부분 일치 검색) 형식의 입력으로 특정 시료를 검색. 그 외 형식은 오류로 안내한다.

### 2. 시료 주문 (접수)

고객 요청에 따라 주문 담당자가 주문을 생성(예약).

- 입력값: 시료 ID, 고객명, 주문 수량.
- 생성 직후 상태는 `RESERVED`.
- 등록되지 않은 시료 ID로는 주문 불가.

### 3. 주문 승인/거절

- **접수된 주문 목록**: `RESERVED` 상태 주문만 표시.
- **주문 승인**: 승인 시 재고 상황에 따라 자동 분기:
  - 재고 충분(재고 ≥ 주문 수량) → 즉시 `CONFIRMED`로 전환.
  - 재고 부족(재고 < 주문 수량) → 생산 라인(생산 큐)에 자동 등록하고 상태를 `PRODUCING`으로 전환. 이때 부족분/실 생산량/총 생산 시간을 계산해 큐 항목에 기록.
- **주문 거절**: 즉시 `REJECTED`로 전환.

### 4. 모니터링

- **주문량 확인**: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASED`) 주문 수 목록. `REJECTED`는 유효한 주문이 아니므로 이 화면에서 제외.
- **재고량 확인**: 각 시료별 현재 재고 수량과, 주문 대비 재고 수량에 따른 상태 표기:
  - **여유**: 주문 대비 재고 충분
  - **부족**: 주문 대비 재고 수량 부족
  - **고갈**: 재고 수량이 0

### 5. 생산 라인

- 생산 라인의 정보를 표시(생산라인 개수, 현재 상태).
- **생산 현황 표기**: 현재 생산 중인 시료의 정보(주문 정보, 현재까지의 생산량 등 — 표기 수준은 자유). 생산 완료 시 해당 주문 상태 `PRODUCING` → `CONFIRMED` 자동 전환.
- **대기 주문 확인**: 생산 큐에 대기 중인 항목을 FIFO 순서로 출력(주문번호, 시료, 주문량, 부족분, 실생산량 등).

### 6. 출고 처리

- `CONFIRMED` 상태 주문 목록 표시.
- 특정 주문에 대해 출고 실행 → 상태가 `RELEASED`로 전환.

## 아키텍처 지침

- 콘솔 기반, 메뉴 선택형 UI. `ConsoleMVC` PoC에서 검증한 Model/Controller/View 분리를 재사용한다.
- 데이터 영속성은 `DataPersistence` PoC에서 검증한 JSON 파일 기반 저장소를 재사용한다(시료, 주문 각각 저장).
- 초기 테스트 데이터는 `DummyDataGenerator` PoC 패턴으로 채울 수 있다.
- 운영 중 데이터 상태 확인은 `DataMonitor` PoC 패턴을 활용할 수 있다.

## 범위 밖 (명세에 없는 것)

- 사용자 인증/권한 분리(주문 담당자 vs 생산 담당자 계정 구분) — 메뉴 단위로만 기능 구분.
- 다중 생산 라인 동시 운영 — 명세는 "하나의 생산 라인은 시료를 하나씩 생산"만 언급, 라인 대수는 예시 UI 상 1개.
- REJECTED 주문의 재처리/복구 흐름.
